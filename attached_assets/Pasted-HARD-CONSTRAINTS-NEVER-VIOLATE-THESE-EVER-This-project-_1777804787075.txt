SUPABASE RULES

Auth:
- Supabase Auth is the source of identity
- Use @supabase/ssr for all Next.js App Router auth integration
- Browser/client components use only:
  - NEXT_PUBLIC_SUPABASE_URL
  - NEXT_PUBLIC_SUPABASE_ANON_KEY
- SUPABASE_SERVICE_ROLE_KEY is server-only. Never expose it in client code, browser
  bundles, logs, docs, tests, or this report
- Remove all hardcoded secrets from tracked files
- Make .env.example safe and complete with placeholder values only

Database:
- DATABASE_URL must always include: ?sslmode=require&pgbouncer=true
  This is mandatory for Supabase Postgres with PgBouncer connection pooling
- Prisma requires a separate direct connection for migrations:
  - DATABASE_URL = pooled connection string (port 6543) with ?sslmode=require&pgbouncer=true
  - DIRECT_URL = direct connection string (port 5432) without pgbouncer
  - prisma.schema must declare: directUrl = env("DIRECT_URL")
  - prisma migrate deploy must always use DIRECT_URL, not the pooled URL
- .env.example must include both DATABASE_URL and DIRECT_URL as safe placeholder entries
- Replit currently manages these secrets:
  - DATABASE_URL (managed by Replit platform)
  - AUTH_SECRET
  - NEXT_PUBLIC_SUPABASE_URL
  - NEXT_PUBLIC_SUPABASE_ANON_KEY
- If SUPABASE_SERVICE_ROLE_KEY is needed for server-only admin operations,
  note it as a required secret but do not hardcode its value anywhere
- Verify DB connectivity on boot
- Run: pnpm prisma migrate deploy (using DIRECT_URL)
- Run: pnpm prisma generate
- Confirm Prisma client initializes cleanly

Session + middleware:
- Supabase session handling must work correctly in Next.js App Router middleware
  and server components
- Private ERP portal routes must be protected server-side, not only via client UI
- Auth boundaries for parent, faculty, admin, and super-admin must be enforced
  on the server
